# 偵測規則：PRD / Pencil / Code 修改如何影響表格勾選

本檔案說明在 Claude 觀察到三產物有改動時，應該如何更新 `SYNC-STATUS.md` 的勾選狀態。

## 修改 PRD.md 的影響

從 git diff 或對話中的 Edit 操作判斷修改類型：

### 1. 可見元素增減（Layout / 視覺結構）

**判斷標準**：PRD 中新增/刪除「畫面元素」描述。例如：

- 新增「在 Home 加入評價按鈕」
- 刪除「Recipes 頁面的搜尋欄」
- 修改「Drinks 卡片從 2 欄改為 3 欄」

**動作**：
- 對應頁面的 **Pencil 欄** → `✗`
- 對應頁面的 **Code 欄** → `✗`
- 備註：`PRD <修改摘要>，待同步`

### 2. Design Token 變動

**判斷標準**：PRD 的「Design System」章節有變動（顏色、字型、圓角、間距）。

**動作**：
- Design Tokens 區塊：對應 token 的 **DESIGN-PROMPT、Tailwind、Pencil** 三欄全 → `✗`
- 因 token 跨頁影響：**所有 Pages 與 Components 的 Pencil 與 Code 欄** → `✗`（除非該頁/元件確認不使用該 token）
- 備註：`Token <name> <舊值>→<新值>`

判斷影響範圍時，可用 Grep 搜尋 token 名稱在程式碼中的使用範圍。

### 3. 純文案 / 描述微調

**判斷標準**：只改了 PRD 中的說明文字、刪減段落，但不增減元素、不改資料模型、不改 token。

**動作**：
- 對應頁面 **Pencil 欄** → `⚠`
- 對應頁面 **Code 欄** → `⚠`
- 備註：`PRD 文案調整，需複查是否影響顯示`

`⚠` 不算不同步，但提醒使用者下次跑 `/sync-status check` 時要看一下。

---

## 修改 .pen 的影響

**動作**：
- 對應頁面/元件的 **Code 欄** → `✗`
- 備註：`Pencil 已更新 <修改摘要>，待 Code 同步`

不動 PRD 欄與 Pencil 欄。

---

## 修改 app/ 或 components/ 的影響

**動作**：
- 該頁面/元件的 **Code 欄** 維持 `✓` 或重新驗證
- 若程式碼修改顯然超出 PRD 描述（例如新增了 PRD 沒提的元素），則 **PRD 欄** → `✗`，備註 `Code 加了 <X>，待補進 PRD`

這個情境較少見，但要記得：「Code 走在前面」也是不同步。

## 偵測時的詢問模板

Claude 觀察到改動後，**不要直接改表格**，先問：

```
你剛剛改了 <檔案>，影響到 <Item>。
要更新 SYNC-STATUS.md 嗎？

[檢查 <Item> / 全表掃描 / 先不檢查]
```

若使用者選「檢查」，套用上述規則。選「先不檢查」就繼續原任務，不要囉嗦。

## 常見誤判

- **PRD 章節重排**：只是把段落順序換了，元素沒改 → 不取消勾選
- **PRD 新增 TODO 註解**：例如 `<!-- TODO: 待設計 -->` → 不取消勾選
- **Code 中的 console.log 或 import 調整**：與 UI 無關 → 不動
- **Pencil 中只改了 frame 名稱**：實際節點屬性沒變 → 不動

判斷不確定時，傾向標 `⚠`（需複查）而非 `✗`，避免噪音過多。
