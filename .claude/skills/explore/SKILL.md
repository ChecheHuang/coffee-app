# Explore

思考夥伴模式。**不寫任何檔案。**

## 範圍

| 會做 | 不會做 |
|------|--------|
| 用 AskUserQuestion 釐清需求 | 寫 SDD 4 件套（→ /propose） |
| Read PRD / 既有實作 | 改 PRD / 改 .pen / 改 code |
| Grep / Glob 校準理解 | 動 SYNC-STATUS |
| 列出選項與權衡 | 建立 openspec change |
| 推薦下一個 skill | 自動跳轉到其他 skill |

## 步驟

### 1. 聽

讀使用者完整問題；不急著回答。識別：
- 是「我想做 X」還是「X vs Y 哪個好」還是「為什麼 X 不行」
- 哪些前提是明說的、哪些是默認的
- 是否有時間/技術/品牌等隱含限制

### 2. 釐清

最多 1–2 題 **AskUserQuestion**，問真正影響後續方向的選擇。避免問可從程式碼/PRD 直接讀出的事。

### 3. 探索

對於每個可能方向：
- 一行描述
- 對應的實際代價（變動範圍、風險、依賴）
- 對比：跟其他方向的差異點

對 BrewMaster Pro 專案，常見維度：
- 是否動到 Design Token（會波及全站）
- 是否新增頁面（會動 PRD §5、Pencil、route）
- 是否需要 Run-and-Verify Gate（任何 UI 改動都需要）
- 是否影響 Onboarding 流程（4 步互鎖）

### 4. 推薦下一步

對話結束時，給一句明確建議：
- 「這方向夠清楚了，可以跑 `/propose` 立案」
- 「還有 X 沒定，先繼續討論」
- 「這應該不是新 change，直接用 `/apply <existing>` 接手既有 change」
- 「先跑 `/three-way-check` 確認現況再決定」

**不要主動呼叫其他 skill**——只給建議。

## Guardrails

- **不寫檔**——任何檔案不動
- **不下決定**——使用者沒同意前不要把討論結果寫進 SDD
- **不要過度提問**——若只需要 1 題就能釐清，就不要問 4 題
- **承認不確定**——不知道答案時就說「不知道」，不要編
- **避免無謂選項**——多個方向都差不多時，直接說「這幾個差異不大，建議直接 /propose」
