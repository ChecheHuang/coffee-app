## Context

- 設定頁「維護」區塊目前三列職責不清：引導式清潔是「執行入口」、維護中心/耗材紀錄是「佔位頁」。本 change 把「執行入口」往下收到維護中心，讓設定頁只保留「進入子區」的導航列。
- 機器狀態詳情頁（`add-machine-status-detail`）已定義「維護提醒」區塊——本 change 的維護中心是該提醒的**動作端**：提醒負責「該做什麼」，維護中心負責「如何執行」。兩者分工明確。
- 既有引導式清潔 Alert 流程（「確定執行」→「清潔中」）其實是半成品狀態機。本 change 把它正規化為通用的「動作卡 → 確認 → 執行中 → 完成」流程，套用到四個維護動作上。

## Decisions

### 兩頁共一個 change

兩頁邏輯上分離（維護 vs 耗材），但：

- 共用「維護主題」設計語言，Pencil 畫一次可共用 tokens。
- 共用一份 PRD 更新（§5.17 + §5.18 連號，設定頁調整一併）。
- 耗材補充本身是一種「維護動作」，將來可能互相 deep link。

拆成兩個 change 的代價：重複跑 /apply Phase B（畫稿同步）兩次，不划算。

### 扁平路由：`/maintenance` + `/consumables`

不採用 `/maintenance/consumables` 層級路由的理由：

- 兩頁是平行概念，不是父子關係。
- 設定頁導航直接對應 route，閱讀程式碼時不需要額外推理。
- 未來若要加「維護動作詳情」，使用 `/maintenance/[action]` 更自然，不會被 consumables 佔位。

### 引導式清潔移進維護中心

使用者於 /explore 階段確認此決策。額外理由：

- 保留獨立列會讓「維護中心」頁與設定列功能重複。
- 統一四動作走同一流程（確認 → 執行 → 完成），UI/UX 一致。
- 設定列變少、層次更清楚。

### `executeAction` 的 mock 行為

本階段：

1. 更新 store 的 `lastExecuted` 為 `Date.now()`
2. 彈出 Toast「[動作名稱] 執行中…」，約 2 秒後改為「[動作名稱] 已完成」

不做：

- 真實的長時間進度條（會增加狀態機複雜度）
- 多步驟引導畫面（引導式清潔原本可能往這方向——本階段先統一為 Toast，未來需要時再另案做成動作詳情頁）

### Toast 元件選型

目前專案無現成 Toast 元件。選項：

1. **內建**：`ToastAndroid` + iOS Alert fallback — 體驗割裂但零依賴
2. **第三方**：`react-native-toast-message` — 跨平台一致但多一個依賴
3. **自製**：Reanimated 實作簡易 Toast — 可控但工作量大

`/apply` 階段依當下時間壓力決定。建議優先自製（能與 dark luxury theme 一致），若時程緊用方案 1 過渡。

## Risks / Open questions

- **耗材估計換期演算法**：mock 資料中「約 X 天後更換」是隨手編還是有基準？建議在 store 初始化時以「剩餘量 ÷ 平均每日消耗」粗估（平均每日消耗寫在 mock 資料即可），讓數字看起來有依據。
- **引導式清潔原本的二層 Alert 行為保留**：設定頁列移除後，原來的「確定執行 → 清潔中」Alert 流程將透過維護中心重現。需確保使用者體驗連續——用維護中心的標準流程取代，而非簡單刪除。
- **耗材歷史列表資料量**：mock >=5 條能看，但實際使用者若幾個月才換一次，歷史頁會很稀疏。本階段不處理，使用者回饋後再調。
- **設定頁移除一列影響整體佈局**：「維護」section 從 3 列變 2 列，視覺會變矮。/apply 階段留意是否需要在 Pencil 調整 section spacing。
- **與機器狀態詳情頁的 deep link**：提醒 → 維護中心的導航（例：「除垢到期」→ 點擊帶參跳到除垢動作卡）屬於本 change out of scope，但 `/apply` 實作時若順手，可以在維護中心支援 `/maintenance?highlight=descale` 這類 query 參數，為未來留接口。
